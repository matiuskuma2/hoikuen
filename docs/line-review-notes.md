## 1. レビュー経緯

### 2026-03-04: 初回レビュー実施

LINE Schedule Collection System v2.0 の設計レビューを実施。
以下の方針で整理:
- **事実と推測を分離** して事故を防ぐ
- **計画のみ（コード変更/commitなし）** のルール適用
- 誤った推定仕様・不正確なdiffを排除

### レビューで採用OKとなったポイント

1. **事実確認（F1〜F9）は正しい方向**
   - `schedule_plans` に `breakfast_flag` がない
   - Python Generator に `has_breakfast` と ¥150 が実装されている
   - `seed.sql` と Python Generator の time_boundaries がズレている
   - `charge_lines` CHECK制約に `breakfast` がない（テーブル再作成が必要）

2. **推定禁止スタンスは必須**
   - 食事推定（朝食/夕食）は誤請求の温床
   - 変更締切は園運用確認なしで決めない

3. **確認①②③の3点がクリティカル**
   - 保存先（直接UPSERT vs 仮登録/確定）
   - 締切/当月変更の運用
   - 朝食の要否

### レビューで修正が必要だった箇所

1. **seed.sql diff: `close` を 21:00 にするのは根拠が薄い**
   - `close = 通常保育終了` であり、延長帯は別概念
   - 修正版: `extension_start=20:00`, `night_start=21:00`, `close=20:00`（変更なし）

2. **LINE SDK非対応は断定しない**
   - 正: Workers では Node依存SDKが動かない可能性がある
   - 正: 安全策として署名検証＋HTTP直叩きで実装する
   - 誤: 「LINE SDKはCloudflare Workers非対応」（断定）

3. **夕食/延長/夜間は別概念として分離が必要**
   - 夕食提供の閾値（何時から提供されるか）
   - 延長課金の閾値（20:00〜21:00）
   - 夜間保育の閾値（21:00以降）
   - これらを混同すると事故る

## 3. 決定会議用1枚資料（2026-03-06時点）

### 目的
保護者がLINEで「来月の利用予定」を提出 → AIが不足項目を聞き返し → 確定 → 既存システムの予定/ダッシュボード/帳票に反映。
※推定で埋めない（誤請求防止）。

### 現状の事実（コード根拠あり）
- `schedule_plans` に `breakfast_flag` がない（Python generatorは `has_breakfast`/¥150 を既に実装）
- `seed.sql` の time_boundaries と Python generator の time_boundaries が不整合
- `charge_lines` CHECK制約に `breakfast` がなく、追加にはテーブル再作成が必要
- ロック/締切（確定）運用は現行要件・DBに未定義

### リスク（決めないと設計が固まらない）→ 全て解消済み
- ~~締切/当月変更の範囲~~ → 確定（②）
- ~~朝食~~ → 確定（③: 提供なし、将来対応）
- ~~夕食/延長/夜間の混同~~ → 分離方針確定
- ~~食事入力の負荷~~ → 確定（④: 全て園側管理、LINE入力不要）

### 修正提案（seed.sql）— 採用前提ではなく案
```
extension_start: "20:00"  (現行 "18:00" → 修正)
night_start: "21:00"      (現行 "20:00" → 修正)
```
